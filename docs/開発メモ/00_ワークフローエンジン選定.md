# Durable Async Function Runtime / Workflow Engine 選定レポート

## 1. 背景

長時間にわたる処理フロー（例：メール送信後、ユーザの返信を数日待つ）を扱うためのワークフローエンジンを検討した。

想定する処理例：

```python
await send_email()
response = await wait_response()
```

このように **通常の Python の async 関数のような書き味で長時間待機を扱いたい**。

また個人利用を想定しており、以下の要件がある。

---

## 2. 要件

### 必須要件

- Python でワークフローを書ける
- 数日など **長時間待機可能**
- 待機中に **プロセスが常駐しない**
- 外部イベントで再開できる

例：

```
send email
↓
数日待機
↓
返信イベントで再開
```

### 追加要件

- **完全セルフホスト**
- **SaaS依存を避けたい**
- できれば **普通の Python 関数のような書き味**

---

## 3. 最初の候補

検討対象として以下のワークフローエンジンを挙げた。

- LangGraph
- Temporal
- Inngest
- Hatchet
- Prefect

---

## 4. 初期比較

| Engine    | 長時間ワークフロー | Python | Event Wait | 備考                |
| --------- | ------------------ | ------ | ---------- | ------------------- |
| LangGraph | △                  | ○      | △          | AIエージェント向け  |
| Prefect   | △                  | ◎      | △          | ETL向け             |
| Temporal  | ◎                  | ○      | ◎          | 本格的              |
| Inngest   | ◎                  | ○      | ◎          | Serverless workflow |
| Hatchet   | ◎                  | ◎      | ◎          | Temporal軽量版      |

**LangGraph / Prefect は用途に合わないため除外**

残り：

- Temporal
- Inngest
- Hatchet

## 5. ワークフロー定義の書き味

### Temporal

特徴：

- workflow class
- activity
- signal

例：

```python
@workflow.defn
class WaitForReplyWorkflow:

    @workflow.signal
    def mark_replied(self):
        self.replied = True

    @workflow.run
    async def run(self):
        await workflow.execute_activity(send_email)

        await workflow.wait_condition(
            lambda: self.replied,
            timeout=timedelta(days=7)
        )
```

#### 特徴

長所

- durable workflow
- 非常に堅牢
- 大規模実績

短所

- workflow/activity/signal など **専用概念が多い**
- Python関数の書き味から遠い

---

### Inngest

特徴：

- event-driven
- step API

例：

```python
async def fn(ctx):

    await ctx.step.run("send-email", send_email)

    reply = await ctx.step.wait_for_event(
        "email.replied",
        timeout="7d"
    )
```

#### 特徴

長所

- serverless workflow
- イベント駆動に強い

短所

- `step.*` APIが必須
- Python関数の自然さは低い

---

### Hatchet

特徴：

- durable task
- child task
- context object

例：

```python
@workflow.durable_task()
async def main(input, ctx):

    await ctx.run_child(send_email_task, input)

    event = await ctx.aio_wait_for(...)
```

#### 特徴

長所

- Python関数に比較的近い
- Temporalより軽量

短所

- `ctx` object が必要
- APIがまだ進化中

---

## 6. 「普通の Python 関数で書きたい」問題

理想：

```python
await send_email()
response = await wait_response()
```

しかし実際の workflow engine では

| Engine   | 必須API     |
| -------- | ----------- |
| Temporal | workflow.\* |
| Inngest  | step.\*     |
| Hatchet  | ctx.\*      |

つまり **ランタイムが workflow state を管理する必要があるため、完全に隠すことは難しい**。

ただし Hatchet は **薄いラッパを作ればかなり隠せる**。

---

## 7. Durable Async Function Runtime という視点

議論の中で本質は

**workflow engine**

ではなく

```
durable async function runtime
```

であると整理された。

これは

```
await wait_event()
```

のような関数を

- 状態保存
- 停止
- イベント到着で再開

できる仕組み。

---

## 8. Durable Wait の本質

長時間待機には以下が必要。

```
state storage
event routing
workflow resume
```

つまり

```
execution_state = f(event_history)
```

このため

**必ずどこかに orchestration 基盤が必要**。

---

## 9. 常駐コスト問題

ユーザ要件：

> 待機中のために常駐サーバを持ちたくない

しかし durable workflow では

最低限

- 状態保存
- イベント受付

が必要。

つまり

**完全に常駐ゼロは原理的に難しい**。

---

## 10. SaaS vs Self-host

### SaaS

例

- Inngest Cloud
- Temporal Cloud

特徴

- 待機状態は SaaS 側
- 自分のサーバ不要

問題

- サブスク依存

---

### Self-host

例

- Temporal
- Hatchet

特徴

- orchestration server が常駐

問題

- インフラコスト

## 12. Postgres 常駐問題

DBを使う場合

```
Postgres
```

自体が常駐サービスになる。

つまり

```
durable runtime
+
Postgres
```

は

**常駐コストが発生する**。

---

## 13. 常駐コストを下げる方法

### 方法1

SQLite + cron

```
SQLite
+

+
polling
```

特徴

- 常駐DB不要
- 即時再開不可

---

### 方法2

小さい常駐プロセス

```

+

```

特徴

- webhook対応
- 常駐コスト小

---

### 方法3

自宅常駐マシン

例

- Raspberry Pi
- NAS

---

## 14. 最終整理

### SaaSなし + durable wait

完全な解は存在しない。

理由

```
durable wait
=
state storage
+
event routing
```

これらは必ず常駐する。

---

## 16. 結論

ユーザ要件

- durable async
- Python関数風
- SaaSなし
- 常駐コスト最小

を考慮すると、

最も現実的なのは

### Option 1

**Hatchet self-host**

- Python書き味
- durable workflow

### Option 2

**SQLite + cron + 自作 runtime**

- 最小コスト
- 完全セルフホスト

---

## 17. 設計方針まとめ

```
durable workflow
=
state persistence
+
event resume
```

この2つをどう実装するかが本質である。

ワークフローエンジンは

**この問題の既製ソリューション**

であり、

小規模用途では

**簡易 runtime を自作する方が合理的な場合もある。**
