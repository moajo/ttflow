# 01: 開発再開

日付: 2026-03-09

## 経緯

ttflowは「durable async function runtime」を自作したもの。
既存のワークフローエンジン（Temporal, Inngest, Hatchet, Prefect, LangGraph）を検討した結果、以下の要件をすべて満たすものがなかったため自作に至った。

### 要件

- Python でワークフローを書ける
- 数日以上の長時間待機
- 待機中にプロセスが常駐しない
- 外部イベントで再開できる
- 完全セルフホスト（SaaS依存なし）
- 常駐コスト最小

### ttflowの設計方針

`durable workflow = state persistence + event resume` を最小構成で実現する。
orchestrationサーバーを持たず、`client.run()` のワンショット実行で動作する。

## 本日の議論

### 既存エンジンとの比較結果

- **Temporal**: 最も堅牢だが常駐ワーカー必須。要件不適合
- **Inngest**: アーキテクチャ的に最適だがセルフホスト不可
- **Hatchet**: Python SDKがalpha段階
- **Prefect**: データパイプライン向け。イベント待ちパターンが弱い
- **LangGraph**: AI特化。汎用ワークフローに不向き

### ttflowの要件充足状況

- 必須要件はすべて達成済み
- 「普通のPython関数の書き味」は原理的限界の範囲内で達成（`RunContext` 1つの受け渡し）
- 即時再開は `run_by_lambda.py` でwebhook経由で対応可能

### モダナイズ計画の策定

長期間放置されていたため、ツールチェーンと依存関係が古い。
Phase 1〜6 のリファクタリング計画を策定した。→ `docs/refactor_and_modernize.md`
